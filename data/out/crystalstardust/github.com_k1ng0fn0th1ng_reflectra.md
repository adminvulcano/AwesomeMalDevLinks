# Reflectra

[Permalink: Reflectra](https://github.com/k1ng0fn0th1ng/reflectra#reflectra)

![Language](https://camo.githubusercontent.com/5a7be820a14fcad7b5270634bddb04516e7310a383acf0ab03869cdb3f4ea7ec/68747470733a2f2f696d672e736869656c64732e696f2f62616467652f6c616e67756167652d4325323025324625323041534d2d626c7565)![Platform](https://camo.githubusercontent.com/2f7394385674fcab2f518d7c9509bb3c10924f0f5f9b6b66b1d9028d40868534/68747470733a2f2f696d672e736869656c64732e696f2f62616467652f706c6174666f726d2d57696e646f77732532307836342d6c6967687467726579)![Type](https://camo.githubusercontent.com/44d048c03a03cc47c67368926bf72aed044829070c5fbd9faacc8f4992c1e010/68747470733a2f2f696d672e736869656c64732e696f2f62616467652f747970652d5544524c2d707572706c65)![Status](https://camo.githubusercontent.com/3c508c35459ceb6fcb8e5f47a89d0468ff31a76f815be22e83f7924d8856cf6b/68747470733a2f2f696d672e736869656c64732e696f2f62616467652f7374617475732d72657365617263682d626c7565)

Reflectra is a **User-Defined Reflective Loader (UDRL)** built on top of [Crystal Palace](https://tradecraftgarden.org/crystalpalace.html) and inspired by the work of [Rasta Mouse](https://github.com/rasta-mouse/Crystal-Kit).
It focuses on providing a reusable, modular UDRL architecture that can be integrated across multiple C2 frameworks while maintaining full control over execution flow and evasion.

## Overview

[Permalink: Overview](https://github.com/k1ng0fn0th1ng/reflectra#overview)

Reflectra provides:

- Custom reflective loading pipeline built on top of Crystal Palace
- Modular evasion (syscalls, spoofing, patching)
- Brings custom evasion capabilities to Beacons
- Full control over execution flow and memory layout
- Designed for integration with DLL-based C2 payloads

It can be integrated into any framework that delivers payloads as DLLs.

## What is a User-Defined Reflective Loader (UDRL)

[Permalink: What is a User-Defined Reflective Loader (UDRL)](https://github.com/k1ng0fn0th1ng/reflectra#what-is-a-user-defined-reflective-loader-udrl)

A User-Defined Reflective Loader (UDRL) is a custom loader that:

- Manually maps a DLL into memory
- Resolves imports and relocations
- Executes the entrypoint without using the Windows loader

Unlike standard reflective loaders, a UDRL gives full control over:

- Memory layout and permissions
- Import resolution strategy
- Execution flow
- Transformation and obfuscation logic

## Features

[Permalink: Features](https://github.com/k1ng0fn0th1ng/reflectra#features)

### Core

[Permalink: Core](https://github.com/k1ng0fn0th1ng/reflectra#core)

- Manual PE loading (sections, relocations, imports)
- Dynamic DLL entrypoint resolution (`Start`, `Run`, `Main`, etc)
- Loader self-cleanup after execution (Free & Run)
- Debug mode with verbose logging (removed in release builds)

### Evasion

[Permalink: Evasion](https://github.com/k1ng0fn0th1ng/reflectra#evasion)

- Stack call spoofing using [Draugr](https://github.com/NtDallas/Draugr)
- Indirect syscalls using [LibGate](https://github.com/rasta-mouse/LibGate)
- ETW user-mode patching (byte patching)
- DLL masking via XOR encryption

### Build & Pipeline

[Permalink: Build & Pipeline](https://github.com/k1ng0fn0th1ng/reflectra#build--pipeline)

- Integration with Crystal Palace
- Automatic YARA rule generation and mutation dealing with [islands of invariance](https://rastamouse.me/islands-of-invariance/)
- Simple stub generation for standalone execution

### Testing

[Permalink: Testing](https://github.com/k1ng0fn0th1ng/reflectra#testing)

- Tested against Defender-derived YARA rules using [defender2yara](https://github.com/t-tani/defender2yara)

## Install

[Permalink: Install](https://github.com/k1ng0fn0th1ng/reflectra#install)

Reflectra depends on external components that are not redistributed, such as [Crystal Palace](https://tradecraftgarden.org/crystalpalace.html) and Tradecraft Garden libraries.
To install all required dependencies:

```
./install.sh
```

This will:

- Download Crystal Palace distribution files
- Download Tradecraft Garden sources
- Build libtcg locally
- Prepare the environment for building payloads

## Usage

[Permalink: Usage](https://github.com/k1ng0fn0th1ng/reflectra#usage)

### AdaptixC2

[Permalink: AdaptixC2](https://github.com/k1ng0fn0th1ng/reflectra#adaptixc2)

1. Generate your beacon as a DLL

2. Build the payload using Reflectra:



```
./build.sh <dll_path> <output_path>
```

3. Execute using the stub or the `run.x64.exe` from Crystal Palace:



```
.\run.x64.exe out.x64.bin
```


### With Any Other C2

[Permalink: With Any Other C2](https://github.com/k1ng0fn0th1ng/reflectra#with-any-other-c2)

1. Generate a DLL payload (preferably stageless)
2. Build it using Reflectra
3. Execute via:
   - stub executable
   - custom runner
   - integrated loader
4. You will need to modify `spec/pico.spec` and overwrite `TARGET_FUNCTION` to target the function to execute

   - This is required because Reflectra does not assume a fixed entrypoint, allowing flexibility across different C2 frameworks.

### Debug

[Permalink: Debug](https://github.com/k1ng0fn0th1ng/reflectra#debug)

You can debug it using this command:

```
./build.sh --debug ./dist/demo/test.x64.dll <output_path>
```

Debug messages are emitted via `OutputDebugStringA` and can be viewed using `DebugView` from SysinternalsSuite

### Output

[Permalink: Output](https://github.com/k1ng0fn0th1ng/reflectra#output)

- `out.x64.bin` → transformed payload
- `rules.yar` → generated YARA rules
- `stub.exe` → executable wrapper
- `out.x64.debug.bin` → optional debug payload

## Limitations

[Permalink: Limitations](https://github.com/k1ng0fn0th1ng/reflectra#limitations)

- Some DLL payloads (e.g., Sliver) may not be compatible due to their Go-based runtime
- Dynamically resolved APIs are not hooked; only functions resolved via the IAT are affected

## Design Goals

[Permalink: Design Goals](https://github.com/k1ng0fn0th1ng/reflectra#design-goals)

- Bring UDRL to public C2 like AdaptixC2
- Define a flexible structure for Crystal Palace projects that can be used with any project
- Keep the loader minimal and modular
- Separate build-time logic from runtime execution
- Avoid tight coupling with any specific C2
- Enable experimentation and research
